# AI Hall of Shame (AHOS) — Copilot Instructions

> **🚨 CRITICAL: Never Read .env Files**
> 
> **NEVER read, display, print, log, or output the contents of `.env`, `.env.local`, `.env.*.local`, or any file containing live credentials.** This is a hard rule with no exceptions. If asked to debug environment issues, suggest checking variable names only — never read the file content.

## Project Overview
A humorous SPA for crowd-sourcing AI misbehavior stories. Reddit-like forum with flat comments, passkey auth, voting. Deployed on Cloudflare Pages + Workers + D1 at $0/month.

## Tech Stack
- **Frontend**: SvelteKit 2 (Svelte 5 runes) SPA mode (`ssr: false`), Tailwind CSS v4
- **Backend**: SvelteKit API routes (`+server.ts`) running as Cloudflare Workers Functions
- **Database**: Cloudflare D1 (SQLite)
- **Auth**: WebAuthn passkeys (`@simplewebauthn/server` + `@simplewebauthn/browser`), JWT via `jose`
- **Tooling**: Node 24 (volta-pinned), pnpm via corepack, TypeScript 5.9 strict

## Architecture
- Single codebase, single deployment to Cloudflare Pages
- Pages (`src/routes/*.svelte`) are client-side rendered (SPA)
- API routes (`src/routes/api/**/+server.ts`) run server-side as Workers
- No cookies — JWT in `Authorization: Bearer` header, stored in `sessionStorage`
- No SSR — all page rendering is client-side
- D1 accessed via `platform.env.DB` in server routes

## Key Design Decisions
- **Plain text posts only** — no markdown, no `{@html}` (XSS prevention)
- **Dark-only theme** — "Neon Shame" palette (no theme toggle)
- **Flat comments only** — no nested/threaded comments
- **Optimistic voting** — instant UI update, background API call
- **7-day JWT expiry** — stored in sessionStorage + Svelte store
- **Username generation** — `SHA-256(credentialId)` → adjective-noun-number

## Code Conventions
- Use Svelte 5 runes (`$state`, `$derived`, `$effect`) — no legacy `$:` reactive statements
- Use `$props()` for component props — no `export let`
- TypeScript strict mode everywhere
- All DB queries use parameterized prepared statements (`db.prepare(sql).bind(...)`) — NEVER string interpolation
- Input validation with Zod schemas in every API route
- Security headers set via `_headers` (Cloudflare Pages declarative header config — lives at project root, NOT in `static/`)

## The `_headers` File

`_headers` is a [Cloudflare Pages feature](https://developers.cloudflare.com/pages/configuration/headers/) for setting HTTP response headers declaratively. It is NOT a custom file — it's a standard mechanism, like `_redirects`. The `adapter-cloudflare` plugin includes it in the deployment automatically. It sets:
- `Content-Security-Policy`
- `X-Frame-Options`
- `X-Content-Type-Options`
- `Referrer-Policy`
- `Permissions-Policy`
- CORS headers for `/api/*` (allowing `https://hallofshame.cc`)

## Security & Conventions (Critical Rules)

**Development:**
1. NEVER read `.env` files (see callout at top)
2. Never use `{@html}` with user-provided content
3. Never use `export let` — use `$props()` instead (Svelte 5)
4. Never use `$:` reactive statements — use `$derived` / `$effect` instead (Svelte 5)

**Database & API:**
- Always use parameterized queries: `db.prepare(sql).bind(value)` — NEVER string interpolation
- Use whitelist mapping for sort parameters — NEVER interpolate user input into SQL clauses
- Validate all inputs with Zod before processing
- Rate limit all POST endpoints (tiered: heavy 5/user 15/IP, light 30/user 60/IP per minute)
- Rate limit GET endpoints (120/IP per minute)
- Check bans → rate limits → auth in middleware (in that order)
- Use timing-safe comparison for backup code verification
